# 会话反思

> 从每次会话中提取可复用经验，持续改进 AI 编程基础设施。入口: [.claude/_index.md](../../_index.md)

## 核心原则

**信息密度优先** — 删除冗余 > 精简现有 > 新增内容 > 加索引。权威源保留完整内容，其他文件改为一行引用。

**精进过程，不给精确步骤** — 本 skill 给出思考方向和判断依据，不给出精确步骤。精确命令会限制 AI 的可操作空间。

### 最高优先级：理解先于行动

reflecting 的成功标准分两步：

1. **是否真正理解了本次会话的根本性问题？**（因果链清晰，不是"xx 没做好"的表面描述）
2. **基于理解，是否产出了恰当的改进？**（文件编辑 / 行为规则调整 / 新建能力 / 明确声明边界）

**理解优先于行动。没有理解清楚就改文件，比不改更差。**

坏 reflecting：把"找到最近的可编辑目标"当成"找到了根因"——发现 git status 没检查 → 加一条规则，而不是追问为什么整个分析阶段都没做正确的事。

好 reflecting：能说出问题的因果链（A 导致了 B，B 表现为 C），然后针对链上最薄弱的一环做改进。

**写文件不是绩效指标** — 真正理解了一个问题但判断不需要改文件，也是成功的 reflecting。每轮都改一个文件但离根因越来越远，是失败的 reflecting。

**执行纪律**：
1. **先理解，后行动** — 停一轮。确认理解了问题的因果链，而不是找到了最近的可修复目标。问自己：这个改进能防止同类根因重现，还是只修了表层症状？只修症状 → 继续追问
2. 识别 1-3 个最高影响根因
3. **检查 Session 任务追踪** — CLAUDE.md 要求 `/tmp/llm-sdk-session-{date}.md`。文件不存在 → 本次会话漏建，记录为层一改进项
4. 直接 edit 目标文件
5. 展示变更摘要请用户确认
6. 跳过"有道理但不够格写入"的发现

**反思深度自适应**：
- 会话无异常无新发现 → 一句话结束，不强制执行
- 会话有一次明显问题 → 聚焦该问题，改一处基础设施
- 会话暴露出系统性流程断裂 → 改 2-3 处文件

**迭代深化** — 一次反思可能不够深。用户说"不够" → 不是在同一深度上找更精确的修复，是**退一步从更根本的层面重新看**。第二轮应比第一轮更接近根因，第三轮应挑战反思方法本身。禁止每轮都在同一深度绕圈——如果连续两轮改的都是同一层级的问题，说明第一轮就没想清楚。

### 检测信号 → 对应反思维度

| 信号 | 命中条件 | → 维度 |
|------|---------|--------|
| agent 超时/超轮/副作用 | 单 agent ≥15 tool calls 或 ≥10min 或操作导致编译错误 | → 层一：prompt 是否缺关键约束？ |
| 跨 Provider 同 pattern 修复 | ≥2 Provider 相同根因 | → 层二：哪个 gate 应提前捕获？ |
| gate 未覆盖的错误 | gate PASS 但仍有错 | → 层四：gate 定义是否完整？ |
| review BLOCK/HIGH | project-reviewer 发现 BLOCK 或 HIGH | → 层一 + 层二 |
| gate 跳过 | lead 未执行合入 gate 中的步骤 | → 层一：gate 描述是否足够强制？ |
| 单点修复无系统性扫描 | 修复了一个错误实例，但未检查同类 Provider/同类路径 | → 层二：此 pattern 在系统其他位置是否存在？发现一个 = 扫描全系统 |
| AI 降维回答/答非所问 | AI 把复杂问题简化成自己会回答的问题，避开真正的评估维度 | → **能力缺位**：不是 prompt 不够好，是缺对该类问题的认知框架。处置：① 新建 skill/agent 填补能力；② 或明确声明能力边界。**严禁用 process tweak 替代** |
| graphify 规则被反复跳过 | 单会话 graphify hook 触发 ≥3 次但 AI 全部跳过 | → 层一：是 AI 不知道 graphify 能回答当前问题（能力缺位），还是 hook 对目标型读取过于激进（hook 设计问题）？ |
| 用户连续表示反思不够 | 用户 ≥2 次说"局限了""不够深""思考太少" | → **反思方法本身在降维**：可能在用"找文件改"替代"理解问题"。退一步重新审视反思方式 |

## 反思框架

按优先级递进，每层达到目的即可止，不强制全走。

### 层零：情景学习 — 从交互中提取模式

> 不只看失败，也看成功和偏好。从用户 prompt、上下文、会话情景中提取可复用的认知。

关注信号：用户重复强调的偏好 → skill/agent 行为规则增强；某种交互模式效果好 → skill 增强；通过 prompt 理解到的项目认知 → 文档沉淀；用户纠正 AI 的方向 → 行为规则调整。

产物落点：用户偏好 → skill/agent 行为规则；交互模式 → skill 工作流增强；项目认知 → `docs/` 或 `CLAUDE.md`。所有产出限定在 `.claude/` 和 `docs/` 内。

一条一条从会话中提取，不批量。问自己：这次会话我学到了什么关于这个项目/这个用户/有效协作方式的新东西？

### 层一：自我进化 — AI 为什么没做好

> 最优先。AI 行为由 skills/agents/CLAUDE.md 驱动，做不好说明指令有缺陷。

思路：`没做好 → 哪个 skill/agent/规则 驱动了错误行为 → 缺了什么指令`

关注信号：选错工具 → 触发条件不准；输出偏离意图 → 约束不够；重复同类错误 → 缺 deny 规则；漏关键步骤 → 工作流不完整；做不该做的事 → 边界不清；引用过期文件 → 交叉引用失效。

一条一条诊断，不批量。

### 层二：经验捕获 — 为什么没提前发现

思路：`问题 → 应在哪个阶段被捕获 → 现有流程缺了什么 gate`

≥2 次或明显会重复 → 记录。一次性偶然 → 跳过。

### 层三：文档沉淀 — 落到正确位置

写作风格跟随目标文件。根据会话涉及的模块判断落点：
- 接口/Provider 变更 → `docs/architecture.md` / `docs/providers.md`
- 配置变更 → `docs/quickstart.md`
- 通用模式 → `CLAUDE.md` 或 skill references/

### 层四：基础设施审视 — skills / agents / _index.md

审视 `.claude/` 下 AI 基础设施的质量：

- 跨文件是否有重复规则 → 去重，保留权威源
- 引用路径是否有效 → 修正过期链接
- 新增/删除文件后 `_index.md` 是否同步
- skill/agent 触发条件是否仍准确

**不需要每次全覆盖** — 根据会话涉及的范围，审视相关的部分即可。

### 层五：过程审计 — 工具使用复盘（择机）

≥3 次 reflecting 或明显异常会话时运行。回溯 agent/skill/tool 使用：选型是否正确、时机是否恰当、有无重复派发。产出矩阵（名称 + 次数 + 合理性 + 改进建议）。

## 输出方式

1. 从最相关的层开始，不必按序
2. 逐条展示（根因 + 目标文件 + 变更摘要 + 理由）→ 用户确认或跳过
3. 确认的条目写入目标文件
4. 深度匹配会话重要性：重要会话深挖根因，简单会话快速过

## 选择形式

| 形式 | 适用 |
|------|------|
| 增强 skill / agent | 指令不精确或缺失 |
| 写入 CLAUDE.md | 高频复现规则 |
| 更新 _index.md | 交叉引用同步 |
| 跳过 | 一次性 / 已有覆盖 |
